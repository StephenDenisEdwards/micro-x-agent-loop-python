# API Server — Manual Test Plan

Step-by-step walkthrough of every API server feature. Run these from the project root directory.

> **Prerequisites**
> - Python 3.11+ with the agent installed (`pip install -e .`)
> - A working `config.json` with at least one LLM provider
> - `.env` with valid API keys
> - `fastapi`, `uvicorn`, `websockets` installed (included in dependencies)

> **Cleanup between test runs**
> Sessions are in-memory and reset when the server restarts. No database cleanup needed unless broker is enabled.

> **Interactive API docs**
> Once the server is running, browse the full API interactively:
> - **Swagger UI:** http://127.0.0.1:8321/docs — try endpoints directly from the browser
> - **OpenAPI spec:** http://127.0.0.1:8321/openapi.json — raw JSON schema for code generation

---

## 1. Server Startup

### Test 1.1: Start the server (default settings)

```bash
python -m micro_x_agent_loop --server start
```

**Expected:**
- Logs: `Starting API server on 127.0.0.1:8321`
- Server stays running in the foreground
- No errors about missing config or MCP servers

### Test 1.2: Start with custom host and port

```bash
SERVER_HOST=0.0.0.0 SERVER_PORT=9000 python -m micro_x_agent_loop --server start
```

**Expected:** Logs show `Starting API server on 0.0.0.0:9000`. Press Ctrl+C to stop.

### Test 1.3: Start with broker enabled

```bash
python -m micro_x_agent_loop --server start --broker
```

**Expected:** Logs show both the API server starting and broker components initialising (scheduler, channels). Health endpoint includes broker status.

### Test 1.4: Start via `--broker start` (backwards compatible)

```bash
python -m micro_x_agent_loop --broker start
```

**Expected:** Launches the unified API server with broker enabled — same behaviour as `--server start --broker`.

```bash
curl -X POST http://127.0.0.1:8321/api/sessions \
  -H "Content-Type: application/json" \
  -d "{\"session_id\": \"test-session-1\"}"
```

## 4. REST API — Non-Streaming Chat

### Test 4.1: Basic chat

```bash
curl -X POST http://127.0.0.1:8321/api/chat \
  -H "Content-Type: application/json" \
  -d "{\"message\": \"What is 2 + 2? Reply with just the number.\"}"
```

**Expected:**
```json
{
  "session_id": "...",
  "text": "4",
  "errors": null
}
```

The response includes a `session_id` (auto-created if not provided).

### Test 4.2: Chat with explicit session

```bash
curl -X POST http://127.0.0.1:8321/api/chat \
  -H "Content-Type: application/json" \
  -d "{\"message\": \"Remember that my name is Alice.\", \"session_id\": \"chat-test\"}"

curl -X POST http://127.0.0.1:8321/api/chat \
  -H "Content-Type: application/json" \
  -d "{\"message\": \"What is my name?\", \"session_id\": \"chat-test\"}"
```

**Expected:** Second response should mention "Alice" — the session preserves context.

### Test 4.3: Chat with missing message

```bash
curl -X POST http://127.0.0.1:8321/api/chat \
  -H "Content-Type: application/json" \
  -d "{}"
```

**Expected:** 400 with error message about missing `message` field.

### Test 4.4: Get messages after chat

```bash
curl http://127.0.0.1:8321/api/sessions/chat-test/messages
```

**Expected:** Message list containing the conversation from test 4.2.

## 5. WebSocket — Streaming Chat

### Test 5.1: Basic WebSocket streaming

Use `websocat` (or any WebSocket client):

```bash
websocat ws://127.0.0.1:8321/api/ws/ws-test-1
```

Once connected, send:
```json
{"type": "message", "text": "What is 2 + 2? Reply with just the number."}
```

**Expected events (in order):**
1. One or more `{"type": "text_delta", "text": "..."}` — streaming tokens
2. `{"type": "turn_complete", "usage": {...}}` — turn finished with token counts

### Test 5.2: WebSocket ping/pong

While connected:
```json
{"type": "ping"}
```

**Expected:** `{"type": "pong"}`

### Test 5.3: WebSocket with tool use

Send a prompt that triggers tool use:
```json
{"type": "message", "text": "Read the file pyproject.toml and tell me the project name."}
```

**Expected events include:**
1. `{"type": "text_delta", ...}` — initial text (may or may not appear before tool)
2. `{"type": "tool_started", "tool_use_id": "...", "tool": "filesystem__read_file"}`
3. `{"type": "tool_completed", "tool_use_id": "...", "tool": "filesystem__read_file", "error": false}`
4. More `{"type": "text_delta", ...}` — response after reading the file
5. `{"type": "turn_complete", ...}`

### Test 5.4: Session reuse via WebSocket

Connect to the same session twice (disconnect first, then reconnect):

```bash
# First connection
websocat ws://127.0.0.1:8321/api/ws/reuse-test
# Send: {"type": "message", "text": "Remember that my favourite colour is green."}
# Wait for turn_complete, then disconnect (Ctrl+C)

# Second connection (same session_id)
websocat ws://127.0.0.1:8321/api/ws/reuse-test
# Send: {"type": "message", "text": "What is my favourite colour?"}
```

**Expected:** The second connection should recall "green" from the first conversation.

### Test 5.5: Invalid WebSocket message

While connected, send invalid JSON:

```
not json at all
```

**Expected:** `{"type": "error", "message": "..."}` — server does not crash.

## 9. Python SDK

### Test 9.1: SDK health check

With the server running:

```python
import asyncio
from micro_x_agent_loop.server.sdk import AgentClient

async def main():
    async with AgentClient("http://127.0.0.1:8321") as client:
        health = await client.health()
        print(f"Status: {health.status}, tools: {health.tools}")

asyncio.run(main())
```

**Expected:** Prints server status and tool count.

### Test 9.2: SDK non-streaming chat

```python
import asyncio
from micro_x_agent_loop.server.sdk import AgentClient

async def main():
    async with AgentClient("http://127.0.0.1:8321") as client:
        reply = await client.chat("What is 2 + 2?")
        print(f"Session: {reply.session_id}")
        print(f"Reply: {reply.text}")

asyncio.run(main())
```

**Expected:** Prints the session ID and the agent's reply.

### Test 9.3: SDK streaming chat

```python
import asyncio
from micro_x_agent_loop.server.sdk import AgentClient

async def main():
    async with AgentClient("http://127.0.0.1:8321") as client:
        async with client.stream("Count from 1 to 5.") as stream:
            async for event in stream:
                if event["type"] == "text_delta":
                    print(event["text"], end="", flush=True)
                elif event["type"] == "turn_complete":
                    print()

asyncio.run(main())
```

**Expected:** Streams the response token by token.

### Test 9.4: SDK session management

```python
import asyncio
from micro_x_agent_loop.server.sdk import AgentClient

async def main():
    async with AgentClient("http://127.0.0.1:8321") as client:
        session = await client.create_session("sdk-test")
        print(f"Created: {session.session_id}")

        sessions = await client.list_sessions()
        print(f"Active: {len(sessions)}")

        await client.delete_session("sdk-test")
        print("Deleted")

asyncio.run(main())
```

**Expected:** Creates, lists, and deletes a session without errors.

### Test 9.5: Run the example script

```bash
python examples/sdk_example.py
```

**Expected:** Runs the full SDK example — health check, non-streaming chat, streaming chat, and session listing.

## 12. Error Handling

### Test 12.1: Server startup with invalid config

```bash
python -m micro_x_agent_loop --server start --config nonexistent.json
```

**Expected:** Error message about missing config file. Does not start the server.

### Test 12.2: Concurrent session limit

Set `SERVER_MAX_SESSIONS=2`:

```bash
SERVER_MAX_SESSIONS=2 python -m micro_x_agent_loop --server start
```

Create 3 sessions:
```bash
curl -X POST http://127.0.0.1:8321/api/sessions -H "Content-Type: application/json" -d "{\"session_id\": \"s1\"}"
curl -X POST http://127.0.0.1:8321/api/sessions -H "Content-Type: application/json" -d "{\"session_id\": \"s2\"}"
curl -X POST http://127.0.0.1:8321/api/sessions -H "Content-Type: application/json" -d "{\"session_id\": \"s3\"}"
```

**Expected:** The third session either evicts the oldest or returns a capacity error (depends on implementation — the oldest session should be evicted by timeout/LRU).

### Test 12.3: Session timeout

Set `SERVER_SESSION_TIMEOUT_MINUTES=1`:

```bash
SERVER_SESSION_TIMEOUT_MINUTES=1 python -m micro_x_agent_loop --server start
```

Create a session, wait >1 minute, then try to access it.

**Expected:** The session has been evicted. A new session is created on next access.
