# Salesforce APIs and Agentic Integrations for AI Agents

## Executive summary

Salesforce exposes a broad API surface for *data*, *metadata*, and *events*, plus newer AI-focused interfaces (notably the Models API and Agent API under Agentforce). For an AI agent that must **read, reason, and act** on Salesforce, the most robust architecture is usually: OAuth-based authentication (JWT bearer or authorisation-code flows depending on whether actions are system- or user-delegated), REST/GraphQL/UI API for interactive CRUD and schema discovery, Composite REST for transactional multi-step writes, Bulk API 2.0 for large-scale ingest/extract, and Pub/Sub API (or Streaming API where required) for event-driven reactions to Change Data Capture and Platform Events. ## Technical patterns for AI agents accessing and acting on Salesforce

### Authentication and authorisation patterns

**System-to-system agent (no interactive user): OAuth 2.0 JWT bearer flow.** Salesforce documents JWT bearer flow for server-to-server integrations, where the client posts a signed JWT to the token endpoint to obtain an access token. **User-delegated agent (needs ŌĆ£act on behalf of userŌĆØ semantics): OAuth 2.0 web server flow (authorisation code grant).** Salesforce positions the web server flow as the standard approach for integrating a web app with Salesforce APIs when a user grants access. **Device and special-case flows.** Salesforce documents a device flow for IoT-style integration and a username-password flow for special scenarios; these exist but carry different risk profiles and operational constraints. ŅłĆciteŅłéturn7search4Ņłéturn7search14Ņłü

**Interactive reads (low/medium volume, need fast ŌĆ£situational awarenessŌĆØ).** REST API is the general-purpose baseline: query/search, CRUD, and ŌĆ£limitsŌĆØ monitoring via `/limits`. **Transactional writes (small sets, multi-step plans).** Composite REST API is the core pattern to reduce agent ŌĆ£tool chatterŌĆØ: sObject Tree enables creating nested parent-child records under one root, with all-or-nothing semantics for the request. ### Event-driven vs polling loops

An agent architecture should treat polling as a controlled fallback:

- **Prefer Pub/Sub + CDC** when the agentŌĆÖs goals are triggered by record changes (e.g., ŌĆ£when Opportunity stage changes to Closed Won, do XŌĆØ). CDC is designed to reduce periodic exports/imports and repeated API calls. ŅłĆciteŅłéturn1search2Ņłéturn1search3Ņłü
